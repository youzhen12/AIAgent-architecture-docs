# 1. 客户业务目标与技术约束

- 业务目标：
  - 降低大促期间重大事故次数（如大面积失败、风控失守、配置错误）；
  - 提升活动 ROI（库存利用率、转化率、用户满意度）；
  - 缩短事故定位和恢复时间（MTTR）；
  - 降低运营/运维/风控在活动周期的人力压力。
- 边界：
  - 绝不让 Agent 直接操作交易主链路；
  - 所有规则/配置变更必须通过既有审批流；
  - AI 只作为「诊断 + 建议 + 脚本草案」旁路能力。
- 约束：
  - 系统对延迟极为敏感，不能在主请求链路上加任何同步 LLM 调用；
  - 数据（订单、风控日志）高度敏感，需要严格脱敏与访问控制；
  - 现有监控/风控系统已经较完备，应优先复用。

# 2. 场景与用例

- 活动前：配置检查、容量评估、风险预案建议；
- 活动中：实时健康评分、异常流量分析、运维/运营建议；
- 活动后：复盘报告、策略优化建议。

# 3. 模型与 MaaS 选型

- 访问方式：
  - 自建 OpenAI 兼容网关，独立于交易系统部署；
  - 支持接入 GPT-5.x / Claude Sonnet 等大模型，以及 Qwen 等性价比模型。
- 模型选择：
  - 分析任务：使用高能力模型（GPT-5.x / Claude Sonnet）；
  - 常规说明和简单问答：小模型（GPT-5.3 Instant / Qwen3.5-Plus）。

# 4. 工程链路设计

## 4.1 推理

- 触发：
  - 定时任务 / 阈值告警 / 人工触发；
  - 所有调用都在事件/任务层面，绝不阻塞交易路径。
- Prompt & Context：
  - 包含活动配置、实时指标（QPS、错误率、延迟）、库存/订单数据摘要、关键日志片段；
  - System Prompt 强调「只给建议、不直接执行」以及风险优先级规则。

## 4.2 RAG

- 数据源：历史活动数据、配置快照、复盘报告、Runbook；
- 向量库：Qdrant/Milvus；
- 用于：
  - 找到类似活动作为参考；
  - 复用过往的解决方案和 Runbook 片段。

## 4.3 Agents 与工具

- Agents：
  - `HealthCheckAgent`：健康评分 + 风险摘要；
  - `RiskAnalysisAgent`：异常流量模式分析 + 风控规则草案；
  - `OpsSuggestionAgent`：扩容/限流/降级建议草案；
  - `ReportAgent`：复盘报告生成；
  - `OrderExplainAgent`：为客服解释用户为什么没抢到等问题。
- Orchestration：
  - 自研 orchestrator（基于事件总线）触发和调度上述 Agents；
  - 复杂分析内部使用 LangGraph 表达多步流程（如多次检索 + 多轮分析）。
- 工具：
  - 监控查询工具、日志查询工具、风控系统查询工具、配置中心查询工具；
  - 高危工具（规则/配置变更、限流脚本）只输出草案，不执行。

# 5. 评估与测试

- 指标：
  - 大促期间重大事故数量、平均恢复时长；
  - 风控误杀率/漏杀率；
  - 运营/运维满意度（主观）和 Agent 建议采纳率（客观）。
- 测试：
  - 通过历史活动数据回放，离线评估 Agents 的诊断准确性和建议质量；
  - 在沙箱环境中模拟异常流量，验证 `RiskAnalysisAgent` 的表现；
  - 对生成的规则/脚本草案做静态检查和人审。

# 6. 上线与部署

- 部署：
  - Agent 子系统独立部署在旁路环境，所有访问核心系统都通过只读接口；
- 灰度：
  - 先在少数活动中使用，仅供内部风控/运维参考；
  - 累积足够信心后，逐步扩大使用场景；
  - 始终保留人工决策权。

# 7. 风险与演进

- 风险：
  - 错误建议可能引导错误人工决策；
  - 对数据质量高度敏感，错误的监控/日志输入会导致错误结论。
- 演进：
  - 从只读分析 → 辅助生成规则/脚本 → 低风险自动化（如仅在沙箱执行），但不进入交易主链路。

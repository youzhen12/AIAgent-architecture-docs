> 用途：
> - 作为「金蝶 AI 星辰」企业管理 AI 平台的场景级架构说明；
> - 为后续落地同类「企业管理 AI 一体化平台」提供可复用骨架。

# 1. 背景与目标

- 场景简介：
  - 金蝶 AI 星辰是面向央国企及大型企业的「企业管理 AI」SaaS 平台，以企业级 AI 底座（如金蝶 AI 苍穹）为基础，将财务、人力、供应链、采购、制造、全渠道、协同办公等业务域的数字化能力与多智能体深度融合。
- 业务问题：
  - 传统 ERP / EBC 系统在决策支持、业务洞察、跨域协同和体验上存在明显人工依赖，无法充分利用企业历史数据与实时数据做智能决策。
  - 各业务条线智能能力零散（如单点智能报表、OCR、RPA），缺乏统一的 AI 底座和 Agent 编排能力，难以规模化复制与治理。
- AIAgent 目标：
  - 构建统一的企业管理 AI Agent 平台，提供管理助手 Agent（如 Cosmic）与各业务域专属 Agent（财务分析、ESG 报告、合同比审、毛利分析、人岗匹配等）。
  - 支持企业知识与业务数据统一治理，支撑跨域业务流程中多 Agent 协同决策与自动化执行。
  - 在保障安全合规与信创要求的前提下，提升管理效率、决策质量与员工体验。
- 文档目标：
  - 给出金蝶 AI 星辰场景下的整体架构视图与关键组件划分，便于比照本仓库中的通用决策文档进行选型复用。
  - 为后续细化 MaaS 方案、性能优化和运维体系奠定结构基础。

# 2. 业务与使用场景

- 主要用户与角色：
  - 集团 CFO / 财务共享中心负责人：关注集团级财务分析、成本与毛利管理、资金与税务合规等。
  - HRD / 业务 HR：关注人岗匹配、人才画像、招聘与绩效分析等。
  - 供应链与采购负责人：关注供应链风险监控、供应商管理、采购合规与成本控制。
  - ESG 管理负责人：关注 ESG 报告生成、披露合规性与指标追踪。
  - 业务部门经理 / 一线员工：通过 AI 助手完成日常业务查询、流程发起和任务协同。

- 关键使用场景（示例）：
  - 财务智能分析：输入自然语言问题（如「对比本季度与去年同期的毛利率变化，并指出主要差异原因」），系统自动联接多账套数据、生成分析报告并给出管理建议。
  - ESG 智能报告：从多系统与文档源抽取 ESG 相关指标与文本片段，按监管与披露规范自动生成报告初稿，由人工补充与审核。
  - 合同与风险比审：合同上传后，Agent 自动进行条款解析、风险点提示、对标集团政策与外部法规，并给出改写建议。
  - 供应链与采购洞察：结合采购订单、供应商评级、交付记录和外部风险信息，提供供应链风险预警与采购策略建议。
  - HR 智能助手：支持岗位 JD 生成、候选人筛选、人岗匹配与员工发展路径推荐等。

- 典型业务流程（示例）：
  1. 管理者在 Web 端或移动端发起自然语言询问 → 管理助手 Agent 解析意图 → 调度相关业务 Agent（财务、供应链、ESG 等）→ 聚合多 Agent 的分析结果 → 生成多模态报告并支持追问钻取。
  2. 业务人员上传合同或 ESG 报告材料 → 文档解析 Agent 进行结构化抽取 → 知识检索 Agent 对照企业制度与法规库 → 风险评估 Agent 给出评估与修改建议 → 形成审阅结果回写到业务系统和协同工具。

# 3. 整体架构概览

- 一句话描述整体架构风格：
  - 基于企业级 AI 底座与 EBC / PBC 可组合架构，构建「模型与 Agent 层 + 数据与知识中台 + 业务 SaaS 应用 + 开放生态」的多智能体企业管理平台。

- 架构图占位：
  - [ ] 逻辑架构图（建议：展现用户入口、Agent 编排、模型服务、数据中台、业务云、开放生态等层次）。
  - [ ] 部署架构图（建议：结合云原生 gPaaS、信创环境、多租户与安全域划分）。

- 核心组件清单（简要）：
  - 用户入口（Web/移动/消息入口）：
    - 管理工作台、PC Web 端、移动 App、小程序/企业 IM 机器人入口等。
  - Orchestrator / Agent 编排层：
    - 会话管理、任务与流程编排、Agent 路由与协同（如多 Agent 议事、A2A 协议）、记忆管理等。
  - Agent 集合：
    - 管理助手 Agent（Cosmic 类角色）。
    - 财务分析 Agent、ESG 报告 Agent、合同风控 Agent、供应链洞察 Agent、人力资源 Agent 等领域 Agent。
  - 工具层：
    - 数据查询工具（财务、HR、供应链等多域数据服务）。
    - 文档解析与结构化抽取工具（OCR、NLP Pipe、表格解析等）。
    - 工作流与任务执行工具（BPMN 流程服务、RPA、消息/通知、任务单创建等）。
  - 模型与向量库：
    - 通用大模型（中英文对话、代码、文档理解等）。
    - 领域定制模型（财务、合规、中文法务等）。
    - 企业知识向量库（制度、合同模板、历史报告等）与检索服务。
  - 监控与日志：
    - Agent 与工作流调用日志、提示与上下文审计、质量与反馈指标、成本与性能监控等。

# 4. 核心技术选型与原因

> 本节给出「本场景假定/参考选型」，实际项目中可按照本仓库 `aia-agent-architecture/decisions/000x-*` 进行调整。

## 4.1 模型与推理服务

- 选型结论（草案）：
  - 以国内主流大模型服务为主，支持混合模型编排（通用对话模型 + 文档理解模型 + 结构化推理模型），通过统一推理网关接入。
- 选择原因（要点）：
  - 满足政企场景对数据安全、隐私和本地化部署的要求。
  - 支持金融/财务等对数字与表格推理较高要求的业务场景。
- 参考决策文档：
  - `aia-agent-architecture/decisions/0003-model-and-inference-service-selection.md`
  - `aia-agent-architecture/decisions/0006-foundation-models-selection.md`

## 4.2 向量数据库与检索方案

- 选型结论（草案）：
  - 采用支持多租户与 RBAC 的企业级向量数据库，配合关系型数据库（如 金蝶自有数据服务云 / Postgres 系列）构建混合检索（向量 + 结构化过滤）。
- 选择原因（要点）：
  - 需要对企业制度、合同范本、ESG 报告等非结构化知识进行高质量语义检索。
  - 需要在同一检索链路中融合集团、子公司、业务单元等多层级访问控制。
- 参考决策文档：
  - `aia-agent-architecture/decisions/0001-vector-db-selection.md`

## 4.3 Agent 编排框架

- 选型结论（草案）：
  - 基于内部 Orchestrator 或 LangGraph 类图编排框架实现多 Agent 协同与任务编排，与企业 BPM 平台（流程服务云）集成。
- 选择原因（要点）：
  - 支持复杂业务流程中多 Agent 协同与状态管理。
  - 易于与现有 BPMN 流程、RPA、任务系统集成，实现「AI+流程」闭环。
- 参考决策文档：
  - `aia-agent-architecture/decisions/0002-agent-orchestration-framework-selection.md`

## 4.4 可观测性与日志方案

- 选型结论（草案）：
  - 统一接入企业级日志与监控平台（如 Prometheus + Grafana / SaaS 观测平台），对模型调用、Agent 运行、向量检索和工作流执行进行端到端观测。
- 选择原因（要点）：
  - 满足审计、问题定位与体验优化的需要。
  - 支持对高价值业务流程（财务关账、ESG 报告生成等）做重点监控与告警。
- 参考决策文档：
  - `aia-agent-architecture/decisions/0004-observability-and-logging-selection.md`

## 4.5 工程设计重点

- 提示词工程（要点）：
  - 为不同业务 Agent 设计专用系统 Prompt，约束角色、口径与输出格式（报告结构、表格字段等）。
  - 针对财务与合规场景加强「不可编造、必须引用来源」的提示约束。
- 上下文工程（要点）：
  - 使用检索增强生成（RAG）将企业制度、合同模板、历史报告等注入上下文。
  - 对跨会话任务维护「管理会话记忆」，管理者可以连续追问并在多个报告间跳转。
- 驯控工程 / Harness（要点）：
  - 对关键场景设置人审与多阶段确认机制（例如 ESG 报告、重大合同条款变更）。
  - 通过评测集与在线反馈闭环，持续优化 Agent 策略与提示配置。
- 参考决策文档：
  - `aia-agent-architecture/decisions/0005-engineering-design.md`

# 5. 核心组件与流程

- 组件清单与职责（示例）：
  - 管理工作台前端：提供多角色登录、场景入口、对话式界面与可视化报表呈现。
  - Agent Orchestrator：负责意图识别、Agent 选择与调用链路管理、工具调用与错误恢复。
  - 业务 Agent：面向财务、ESG、合规、供应链、HR 等具体业务域的专用智能体。
  - 工具与连接器：封装各业务云（财务云、HR 云、供应链云等）的 API，提供统一调用接口。
  - 模型服务层：封装大模型推理服务，包括路由、配额控制、缓存与降级策略。
  - 知识检索层：面向企业制度、合同、报告等的向量检索与结构化检索。

- 关键流程说明（示例）：
  1. 对话式分析流程：用户 → 前端 → Orchestrator → 意图识别 → Agent 选择 → 数据与知识检索 → 模型推理 → 报告生成 → 回写与通知。
  2. 文档审核流程：文档上传 → 文档解析工具 → 知识检索 → 风险分析 Agent → 生成审阅结果 → 人工确认 → 归档与回写业务系统。

# 6. 数据与存储设计

- 数据类型与来源：
  - 结构化业务数据：总账、明细账、凭证、预算、采购订单、库存记录、人事档案等。
  - 半结构化与非结构化数据：合同 PDF/Word、ESG 报告、会议纪要、制度文档、外部法规与政策文档等。
  - 行为与日志数据：用户操作日志、Agent 调用日志、模型反馈与评分等。

- 存储与索引方案（草案）：
  - 业务系统自带关系型数据库（通过数据服务云暴露统一查询接口）。
  - 专用向量数据库存储企业制度、合同条款、报告内容等嵌入向量，支持 RAG。
  - 对日志与审计数据使用时序数据库或日志平台（如 Elasticsearch + 对应可视化）。

- 数据生命周期（要点）：
  - 按照集团数据治理要求设置不同数据域的留存周期与脱敏策略。
  - 对模型训练/微调与评估使用的数据集单独管理，确保可追溯与可删除。

# 7. 非功能性要求

- 性能与延迟：
  - 管理者交互类分析场景：P95 响应时间控制在数秒级，可通过异步生成完整报告 + 先返回摘要的方式优化体验。
  - 大批量报告生成与离线分析场景：采用批处理与任务队列，避免对在线系统造成冲击。

- 可用性与容错：
  - 关键服务（Orchestrator、模型网关、向量检索等）多实例部署，支持自动故障转移。
  - 对第三方模型服务提供降级策略（切换备用模型、回退到纯查询与报表模板）。

- 安全与合规：
  - 细粒度权限控制，按角色/组织/业务域限制数据与能力访问。
  - 全链路审计日志，对关键决策与报告生成过程可追踪可追责。
  - 满足信创、国密算法与本地化部署要求（如适用）。

- 成本与资源：
  - 对高频场景采用缓存与摘要复用策略，降低重复推理调用。
  - 通过配额与配速控制不同角色与部门的调用成本。

# 8. 风险、边界与取舍

- 已知风险：
  - 大模型在财务与合规场景中可能产生错误或不完整的建议，需要通过人审与多源校验进行约束。
  - 向量检索可能引入不相关或过时的知识，影响决策质量。

- 边界与不做的事：
  - 平台不直接替代企业最终管理决策，只提供决策辅助与报告生成能力。
  - 初期不引入过多自动执行型 Agent（如自动下单、自动记账），而是以「推荐+人工确认」为主。

- 关键取舍说明：
  - 在体验与安全之间，以安全与可控优先，接受部分场景的交互延迟与人工参与。
  - 在通用模型与行业专用模型之间，优先选择在政企场景已验证的稳健方案。

# 9. 实施计划（可选）

- 版本拆分（示例）：
  - V0：单场景 PoC（如财务智能分析 + ESG 报告生成），验证数据接入、RAG 与报告生成链路。
  - V1：扩展到多业务域 Agent，打通若干关键业务流程（例如合同比审 + 审批流程集成）。
  - V2：引入跨场景协同与多 Agent 协作，将平台能力开放给更多内部团队与 ISV。

- 依赖与前置条件：
  - 企业已有 EBC / ERP 云或等价业务系统，具备较完备的数据底座与数据服务层。
  - 完成数据治理与分级分类，明确高敏感数据的访问与脱敏策略。
  - 明确关键场景的业务 Owner 与评估指标，支持持续优化。

> 用于描述「金蝶 AI 星辰」场景下，面向内部/外部系统提供的模型即服务（MaaS）与智能体即服务（Agent-as-a-Service）整体方案。

# 1. 背景与业务目标

- 背景：
  - 随着金蝶 AI 星辰在财务、HR、供应链、采购、ESG 等业务域的落地，平台内部已经沉淀了多套大模型能力（通用对话、文档理解、结构化推理）以及一批高价值业务 Agent（财务分析 Agent、ESG 报告 Agent、合同比审 Agent 等）。
  - 企业内部其他系统与外部生态（ISV、合作伙伴）有需求以统一方式复用这些模型/Agent 能力，而不是各自重复集成模型服务或单点智能应用。

- 业务目标：
  - 对内：为金蝶内部各业务云（财务云、HR 云、供应链云等）提供统一、可治理的模型与 Agent 服务接口，支撑快速集成与复用。
  - 对外：在满足安全合规的前提下，对集团客户与 ISV 开放选定能力（如报表分析、合同比审、ESG 模板生成等），形成「企业管理 AI 能力市场」。
  - 运营层面：实现对模型调用成本、Agent 效果与服务质量的统一观测与优化，为后续商业化（按调用量计费、按能力套餐计费）奠定基础。

# 2. 能力边界与服务对象

- 服务对象：
  - 内部业务系统：金蝶自有各业务云及其扩展应用。
  - 外部企业客户：通过 API/SDK 或应用市场使用部分标准化能力（如智能分析、智能生成）。
  - ISV / 合作伙伴：在金蝶生态中开发/集成应用时复用模型与 Agent 能力。

- 能力范围（第一阶段）：
  - 模型级能力：
    - 通用对话与问答（支持中文主场景）。
    - 文档解析与结构化抽取（合同、报告、制度文档等）。
    - 报表/表格理解与简单数值推理。
  - Agent 级能力：
    - 财务智能分析 Agent：自动生成财务分析报告、毛利分析、费用结构分析等。
    - ESG 报告 Agent：根据预设模板与企业数据生成 ESG 报告草稿。
    - 合同比审 Agent：对上传合同进行条款识别与风险提示。

- 能力边界（明确不做的部分）：
  - 不直接对外提供底层模型的原始推理接口管理权（如温度、最大 Token 等低层细节），对外以「能力 API」或「Agent 能力」形式暴露。
  - 不对外直接开放高风险自动执行能力（如自动记账、自动下单），外部只获得「建议/草稿」级能力，需要业务系统二次确认。
  - 初期不承担「自定义模型训练托管」职责，微调/训练能力由内部平台独立演进，MaaS 仅暴露已经过治理的模型/Agent。

# 3. 服务形态与接入方式

- 整体形态：
  - 提供两类主形态：
    - 模型服务 API（Model API）：用于内部系统按场景调用抽象好的模型能力（对话、文档解析、RAG 检索等）。
    - Agent 服务 API（Agent API）：面向高阶场景，直接调用业务 Agent，简化调用方逻辑。

- 对内接入方式：
  - 服务发现：通过内部服务网格或 API 网关（如内部 APIGW）暴露统一域名与路径规范。
  - 鉴权与配额：
    - 通过内部统一鉴权体系（如 OAuth2/JWT + 应用凭证）进行调用方认证。
    - 为每个业务云设置配额与限流策略，结合成本中心进行计费/成本核算。
  - 调用模式：
    - 同步调用：适用于对话类、短分析类场景。
    - 异步任务：适用于 ESG 报告生成、长文分析、批量合同审核等长耗时场景，通过任务 ID 轮询/回调获取结果。

- 对外接入方式（面向企业客户与 ISV）：
  - 形态 1：RESTful API + SDK
    - 提供 REST/HTTPS API，配套多语言 SDK（Java/JavaScript/Python 等）。
    - 使用 API Key + 子账号机制进行鉴权，支持按租户/应用维度限流与计费。
  - 形态 2：应用市场与插件化
    - 在金蝶应用/能力市场中，以「能力组件」形式上架典型 Agent 能力（如 ESG 报告生成），由客户在配置界面中完成授权与参数配置。
  - 形态 3：工作流节点/插件
    - 将 Agent 能力封装为 BPM/工作流节点，支持客户在可视化流程设计器中拖拽使用。

- 标准化接口设计原则：
  - 请求体统一包含：调用场景 ID、调用方信息、业务上下文（如组织/账套/时间范围）、用户自然语言输入或结构化参数、可选的输出格式偏好。
  - 响应体统一包含：结构化结果（JSON）、可读文本摘要、引用数据来源列表（数据血缘）、调用成本与耗时指标，便于审计与优化。

# 4. SLA 与非功能指标

- SLA 指标（示例目标）：
  - 可用性：核心 MaaS/Agent 服务年度可用性 ≥ 99.5%。
  - 延迟：
    - 对话类接口：P95 响应时间 ≤ 3s（不含客户网络延迟）。
    - 报告生成类接口（异步）：任务排队时间 ≤ 30s，90% 任务在 3 分钟内完成。
  - 成功率：有效请求的成功处理率 ≥ 99%。

- 性能与扩展性：
  - 水平扩展：所有对外服务通过容器化与服务网格支撑横向扩容，支持大规模并发调用。
  - 缓存策略：对高频重复问题、固定模板报告使用缓存/结果复用，减小模型调用压力。

- 安全与合规：
  - 数据隔离：严格的租户隔离机制，防止数据跨租户泄露。
  - 审计能力：保留足够粒度的调用日志与上下文引用记录，支持事后追溯和合规审计。
  - 内容安全：对模型输出进行敏感信息过滤与内容合规检测（尤其在对外场景）。

- 运维与观测：
  - 全链路追踪：从 API 网关、Orchestrator、模型网关到向量检索与业务系统调用的链路追踪。
  - 指标观测：QPS、延迟、错误率、各模型/Agent 的调用量与成本、租户维度的使用情况。
  - 报警策略：
    - SLA 指标异常（可用性、延迟、错误率）。
    - 单租户/单接口流量异常（例如突增、攻击或误用）。

# 5. 关键设计决策

> 本节聚焦若干对架构和运维影响较大的设计点，方便与 `aia-agent-architecture/decisions/000x-*` 进行比照或沉淀为独立 ADR。

- 决策 1：统一模型网关 vs 服务各自直连模型厂商
  - 结论（建议）：采用统一模型网关（可自建或使用金蝶 AI 底座内置服务），所有 MaaS/Agent 流量统一经过网关。
  - 理由：
    - 便于集中管理不同模型厂商与版本的路由、配额与成本。
    - 支持无感切换模型（升级/降级）并对调用方保持接口稳定。

- 决策 2：模型级接口 vs Agent 级接口暴露边界
  - 结论（建议）：内部可以同时暴露模型级与 Agent 级接口，对外以 Agent 级能力为主，辅以少量模板化模型级接口。
  - 理由：
    - 内部团队可能有更复杂的编排需求，需要直接控制模型调用细节。
    - 外部客户更需要「业务能力」而非「大模型原子能力」，Agent 封装可以屏蔽复杂度并统一体验。

- 决策 3：在线生成 vs 预生成/缓存
  - 结论（建议）：
    - 对话类场景以在线生成为主，辅以语义缓存。
    - 报表/报告类场景（如 ESG 报告、月度财务分析）优先采用任务预生成或定时生成方案。
  - 理由：
    - 控制成本和延迟，同时避免每次调用都从头生成相似内容。

- 决策 4：多模型策略（Routing/Ensemble）
  - 结论（建议）：
    - 对不同任务采用不同模型（Routing），例如通用任务用通用模型，表格推理用专长模型，法务/合规用专用模型。
    - 在高价值场景（如合同比审）可选用模型对比或多模型交叉验证策略。
  - 理由：
    - 兼顾成本与效果，同时在关键场景降低单一模型错误的风险。

# 6. 实施与运维

- 实施步骤（建议）：
  1. 梳理并标准化现有业务 Agent 与模型能力，定义统一能力目录与接口规范。
  2. 建设统一模型网关与 Agent 服务层，对现有应用做最小改造接入。
  3. 选择 1–2 个内部业务云作为首批接入方，验证调用稳定性与业务价值。
  4. 在内部稳定运行后，逐步开放给更多内部门户与外部客户，建立计费与配额管理机制。

- 运维职责划分：
  - 平台团队：负责模型网关、Agent 服务层、API 网关与观测系统的建设和维护。
  - 业务线团队：负责自身使用的能力配置、Prompt 策略和业务逻辑集成。
  - 安全与合规团队：负责内容安全策略、审计规范与定期检查。

- 持续优化机制：
  - 建立反馈通道：收集调用失败案例、业务投诉与改进建议。
  - 建立评测与 AB 测试机制：针对关键场景评估不同模型/Agent 组合的效果与成本，并逐步优化默认策略。

# 今天的想法：我的 Harness Engineering 开发流程

这篇文档记录我现在实际采用的「驭控式」开发模式，可以看作是
`docs/aia-agent-architecture/decisions/0005-engineering-design/harness-engineering.md`
在我个人工作流里的一个具体化版本。

## 1. 我的实际开发流程

1. 用 skill 写技术方案
   - 先通过一个专门的 skill 生成技术方案草稿。
   - 在这个阶段就会给出一组「限制 / 规范 / 前提假设」，并在方案里明确讨论。

2. 评审技术方案
   - 对技术方案进行人工/Agent 评审，看是否合理、是否遗漏关键约束。
   - 评审通过后，这份技术方案就成为后续所有实现的「约束来源」。

3. 基于 openspec 开发代码
   - 使用 openspec 来驱动代码实现。
   - 在调用 openspec 时，需要把前面技术方案中的限制、规范显式注入进去，让生成逻辑在这些约束之内工作，而不是从零开始瞎猜。

4. 开发过程中每一步都通过检测
   - 实际编码时，每完成一小步，就跑一次：
     - pre-commit 检测（lint、格式、基础约束）。
     - test 检测（单元测试 / 集成测试等）。
   - 不通过就回到当前这一步修正，确保「小步前进、每一步都是干净的」。

5. 按照技术方案编写和执行自动化测试
   - 功能全部写完后，会根据技术方案中描述的行为和边界，补充 / 编写对应的自动化测试。
   - 这些测试相当于「把技术方案中的预期行为编码成可执行约束」。

6. 使用 review skill 做代码 Review
   - 开发完成后，不是直接结束，而是通过一个专门的 review skill 做代码 Review。
   - review 的重点对照三类东西：
     - 事先约定的限制和规范。
     - 技术方案本身（实现是否真的遵守了方案）。
     - 代码质量和结构（可维护性、可读性等）。

## 2. 和 harness-engineering 文档的相同点

对比 `harness-engineering.md` 里的定义，我的这个流程本质上也在做「驭控工程」，相同点包括：

1. 约束不是只放在 Prompt 里
   - 技术方案 + 自动化测试 + pre-commit + test 检测，形成了多层次的硬约束。
   - 这些约束和检查本身都是确定性的（linters、测试），而不是依赖模型自己「自觉」。

2. 把「环境 + 工具链」当成控制手段
   - 使用特定的 skill（写方案、review）、特定的开发入口（openspec）、固定的检查工具（pre-commit、测试），构成一个封闭的工作环境。
   - AI/Agent 只是这个环境中的一个参与者，而不是拥有无限自由度的主体。

3. 强调反馈回路
   - 每一步不通过 pre-commit / test 就不能继续，相当于在流程节点上内建了反馈回路。
   - 技术方案 → 实现 → 测试 → review 之间形成了闭环。

4. 「技术方案」作为系统的事实来源之一
   - 类似文档中强调的「代码仓库 + 设计文档是 system of record」，这里技术方案被视为后续所有动作的约束来源，并通过测试和 review 被持续引用。

## 3. 一些区别和补充

1. 关注层级不同
   - `harness-engineering.md` 更偏向系统级 / 平台级视角：
     - 如何为 Agent 设计运行时边界、权限模型、架构约束、巡检 Agent 等。
     - 更关心「一个 Agent 团队长期演化」的质量和熵控制。
   - 我的流程聚焦在「个人 / 小团队的具体开发循环」：
     - 以一次具体需求为单位，从方案到实现到测试到 review 的完整闭环。
     - 更像是把驭控工程落在每次任务上的「微观实践」。

2. 我现在的流程还没有显式的「垃圾回收 / 熵控制」部分
   - 文档中强调要有巡检 Agent、黄金原则、AI 残渣清理等长期治理手段。
   - 我的流程目前主要保证「这一次开发」是干净、可控的，但还没有专门机制：
     - 定期清理过时文档、规则和坏模式。
     - 通过巡检自动发现历史遗留问题。

3. 「策略 / 编排 / 工具 / 模型」的拆层程度不同
   - 文档中建议把策略、编排、工具、模型清晰拆层。
   - 在我的实践里：
     - 策略和编排目前更多是隐含在「我自己怎么用这些 skill 和工具」里。
     - 真正结构化的，是工具层（skill、openspec、pre-commit、测试）。
   - 未来可以考虑把这些策略和编排显式写成配置或文档，让其他人/Agent 也能照着跑同一套流程。

4. 多 Agent 协作程度
   - 文档中设想了多种专职 Agent：主流程 Agent、巡检 Agent、审批 Agent 等。
   - 在我的流程里，目前主要是：
     - 写方案的 skill。
     - 做代码 review 的 skill。
   - 还没有专门的「巡检 Agent / 质量守护 Agent」，更多是我自己在流程中扮演多个角色。

5. 运行时限制的粒度
   - 文档里强调对执行环境做更细粒度的限制（命令白名单、路径限制、生产环境保护等）。
   - 我的流程目前主要通过工具和流程来限制，而不是在系统层面做强隔离：
     - 比如通过 pre-commit / test 阻止低质量或不符合规范的提交。
     - 但对于「能不能执行某个危险命令」这类问题，目前还没有在工具层完全封死。

---

后续如果这个个人开发流程稳定下来，可以再往前走一步：

- 把「怎么用 skill 写方案、怎么用 openspec、什么时候必须跑哪些检查」写成显式规则或脚本，让 Agent 也能照着跑。  
- 引入轻量级的巡检 Agent，专门在这套流程之上、跨多次任务做质量守护。

# 1. 客户业务目标与技术约束

- 客户业务背景：
  - 多渠道客服体系（Web/App/IM/电话）+ 工单系统 + CRM；
  - 服务对象包括 B2C 终端用户和 B2B 企业客户。
- 业务目标：
  - 自助解决率（无需人工介入）≥ 60%；
  - 人工客服平均处理时长（AHT）降低 20%；
  - 用户满意度（CSAT）提升 5pp 以上；
  - 知识库维护和培训成本明显下降（以新客服上手时间衡量）。
- 业务边界与不做的事：
  - 不在当前阶段完全替代人工客服；高价值/高风险场景保留人工兜底；
  - 不直接做账户敏感操作（如资金划转、销户），只提供解释和结构化信息；
  - 不强行接管所有渠道入口，允许客户逐步迁移比例。
- 技术与组织约束：
  - 已有监控与日志平台，可集成 OTel；
  - 客服系统和 CRM 多为内部私有部署，数据出境需严格审批；
  - 客服/运营团队具备基础数据分析能力，但缺乏大规模 LLM 工程经验。

# 2. 场景与用例拆解

- 核心场景：
  1. FAQ 问答；
  2. 工单/订单/设备状态查询；
  3. 升级与投诉处理辅助；
  4. 质检与合规监控。

- 典型用例（示例）：
  - FAQ：
    - 「这个套餐包含哪些服务？」；
    - 「如何修改密码？」；
  - 工单/订单：
    - 「我的工单现在处理到哪一步了？」；
    - 「订单为什么被取消？」；
  - 升级与投诉：
    - 「我要投诉这个处理结果」→ Agent 生成结构化升级单；
  - 质检：
    - 自动抽取对话，检测敏感词与服务规范执行情况。

- 优先级与波次：
  - v1：FAQ + 工单查询（只读）；
  - v2：升级单生成 + 质检建议；
  - v3：更深入的多轮诊断与主动提醒。

# 3. 模型与 MaaS 能力选型

- 模型访问方式：
  - 自建 OpenAI 兼容网关：
    - 内部部署或 VPC 内托管，避免明文数据出公网；
    - 支持多厂商模型接入。

- 模型选择：
  - 对话主模型：
    - 中文/多语言客服为主，优先选 Qwen3.5-Plus / GPT-5.x / Claude Sonnet 中表现较好者；
  - Embedding：
    - 选中文表现好的 Embedding 模型（Qwen / OpenAI）；
  - 不在初期做自有微调，靠 Prompt + RAG + 示例引导。

- 路由策略：
  - FAQ 和简单查询 → 小模型；
  - 复杂投诉与升级摘要 → 大模型。

# 4. 工程链路设计

## 4.1 推理链路

- 入口：渠道接入系统 → 客服网关 → AI Gateway；
- 会话管理：
  - 按用户 + 渠道 + 会话 ID 绑定上下文；
  - 对长对话做摘要压缩（避免 token 爆炸）。

## 4.2 微调

- 初期：不采用微调；
- 后期：可考虑将优质客服对话与质检样本用于指令微调，提升语气与业务合规性。

## 4.3 RAG

- 数据源：FAQ、产品文档、政策条款、系统说明、工单摘要等；
- 向量库：
  - 初期：Postgres + pgvector；
  - 中期：迁移到 Qdrant；
- 检索：按业务线/渠道/语言过滤；FAQ 直接 RAG，复杂问题结合工单/订单数据。

## 4.4 Agent 与工具

- Agents：
  - `RouterAgent`：意图识别与路由；
  - `FAQAgent`：RAG 问答；
  - `TaskAgent`：调用工具访问工单/订单/设备系统；
  - `EscalationAgent`：生成升级单和总结；
  - `QCInspectorAgent`：对历史对话进行自动质检。
- Orchestration：
  - 使用 LangGraph 建模 FAQ/工单/升级流程；
- 工具层：
  - 只读工具（工单/订单/设备查询） + 写工具（仅生成升级单，不直接执行业务写操作）；
  - 高危操作一律 require 人审。

# 5. 评估与测试

- 指标：
  - 自助解决率 / 升级率 / 人工接管率；
  - AHT / 首次响应时间；
  - CSAT / NPS；
  - 知识命中率 / 错误回答率。
- Eval 集：
  - 从历史对话抽取典型问题 + 标准回答；
  - 特别标记高风险问题（合约条款、赔付策略）。
- 测试策略：
  - 离线回放 + Judge LLM + 人审；
  - CI/CD 前在 Eval 集上做回归；
  - 上线后持续监控，抽样人工复审。

# 6. 上线与部署

- 灰度策略：
  - 从单一渠道/业务线开始（例如 App 内 FAQ）；
  - 初期仅作为「建议」提供给人工客服（双列界面），稳定后逐步放给终端用户；
- 安全：
  - 敏感信息脱敏；
  - 所有生成内容与工具调用记录在审计日志中；
- 运营：
  - 客服/运营团队参与调参与知识维护，形成定期复盘机制。

# 7. 风险与演进

- 风险：
  - 回答错误导致投诉增加；
  - 模型 hallucination 产生虚假政策/承诺；
  - 知识库未更新导致回复过期。
- 缓解：
  - 对高风险问题设置「必须升级人工」规则；
  - 显式引用知识来源，避免无证据回答；
  - 知识更新流程与产品变更同步。

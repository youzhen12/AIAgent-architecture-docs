# 描述

Semantic Kernel（简称 SK）是微软开源的一套 AI 应用编排框架，提供「插件/技能（Plugin/Skill）」「Planner/Step」「Prompt 模板」「记忆（Memory）」等抽象，支持 C#、Python、Java 等多语言 SDK。其定位更接近「通用 AI 应用编排 + 插件系统」，在设计 Coding Agent、业务助手等场景时可以作为多 Agent/工作流的底层框架之一，也可以与现有微服务、消息队列等基础设施结合使用。

与 LangGraph 相比，SK 更偏向「技能/插件 + 计划器」的思路，强调将现有系统能力封装为 Plugin，再通过 Planner/Step 组合调用；在微软生态（.NET、Azure）里集成度较高。

# 优点

- 生态与语言支持友好：
  - C#/Python/Java 多语言 SDK，便于在不同技术栈中统一使用同一套编排抽象；
  - 在 .NET 与 Azure 生态中集成度高，适合已有大量 C# 服务的团队。
- 插件/技能模型成熟：
  - 把业务能力封装为「插件/技能」，结合函数调用与元数据，利于将现有 API/服务抽象为可被模型调用的能力；
  - 与 OpenAI 函数调用、Chat Completion、Assistants 等接口形态对齐较好。
- Planner 与步骤式工作流：
  - 提供 Planner/Step 抽象，支持根据自然语言目标自动构造任务计划，再映射到插件调用序列；
  - 适合结构化不强、但可以被拆成多步调用的任务。
- 易于与企业基础设施集成：
  - 官方示例与文档中包含大量与 Azure、Key Vault、存储服务、消息系统等的整合实践；
  - 在日志、配置、鉴权等方面更容易贴合既有基础设施。

# 缺点

- Agent 视角抽象相对薄：
  - SK 的核心更偏向「技能 + 工作流」，对「多 Agent 编排」「可视化状态图」的支持不如 LangGraph 这类专门的 Agent 框架；
  - 如果需要细粒度表达「子 Agent runtime」「前台/后台 Agent 生命周期」，往往需要额外自研控制层。
- 工作流表达能力需要自行约束：
  - Planner 自动生成的计划在复杂场景下稳定性有限，实际项目中常常需要人为限制规划空间或改用手写流程；
  - 缺少统一的「图式」表示方式（相对于 LangGraph 的有向图模型），大规模复杂流程的可视化与复盘需要自己搭建。
- 社区关注点较分散：
  - SK 面向的是广义「AI 应用」，很多最佳实践集中在插件封装、RAG、Azure 集成等，对「多 Agent / Harness 工程」的系统化资料相对较少。

# 适用场景与使用方式建议

- 适用场景：
  - 团队以 C#/.NET 为主，希望在现有后端中引入 Agent 功能，而不想额外引入一套完全独立的编排引擎；
  - 需要将大量现有 API/服务封装为「AI 可调用插件」，并通过 Planner 构造多步调用；
  - 以 Azure 为主的云环境，希望尽量使用微软官方生态中的组件与集成方案。

- 使用方式建议：
  - 在 AIAgent 系统中，SK 可以作为「插件封装 + 轻量工作流引擎」，将业务能力暴露给上层 Agent；
  - 若需要 LangGraph 式的复杂 Agent 图，可考虑：
    - 将 SK 封装为 LangGraph 节点，或
    - 在自研 orchestrator 中以 SK 为「插件执行层」，上层仍用事件流/状态机描述整体 Agent 协作。

# 1. 目标

- 将「记忆」从隐式依赖对话历史/日志的副产品，提升为可设计、可调优、可观测的工程能力。  
- 明确区分并工程化管理不同层级的记忆：
  - 短期工作记忆（per-session / per-task）。  
  - 中期项目记忆（per-project / per-repo）。  
  - 长期用户/团队记忆（偏好、惯例、决策史）。  
- 在有限的上下文窗口和成本约束下，使 Agent 能够：
  - 持续对齐用户/团队的工作方式与偏好。  
  - 记住重要决策和中间结果，减少重复工作。  
- 在安全和可控前提下，避免「记太多」引发隐私泄露、错误固化和行为僵化。

# 2. 记忆工程 vs 上下文工程

记忆工程与上下文工程紧密相关但关注点不同：

- 上下文工程（Context Engineering）：
  - 关注「这一轮推理需要知道什么事实」。  
  - 聚焦采集、切分、索引和注入。

- 记忆工程（Memory Engineering）：
  - 关注「哪些信息值得长期保留，并如何在后续任务中重用」。  
  - 更像是「跨会话、跨任务的状态设计与管理」。

在 Claude Code 的架构中，这种区分体现为：

- 上下文通过 prompts.ts、工具结果、任务描述等动态组装。  
- 记忆通过 `src/memdir/`、session guidance、prompt cache boundary 等机制持久化，并在新任务中被重新注入 system prompt 的特定位置。

# 3. 记忆层次划分

## 3.1 短期工作记忆（Session Memory）

定义：

- 单次会话或单次任务生命周期内的状态：
  - 最近交互历史。  
  - 当前任务的计划、子任务进度、局部结论。  
  - 临时变量（路径、文件、上下文选择等）。

工程要点：

- 不必全部长期存储，但需要：
  - 在会话内高效压缩与检索（会话树、摘要节点等）。  
  - 在必要时提炼出「可升级为中期/长期记忆」的部分（例如：用户偏好变更）。

## 3.2 中期项目记忆（Project / Repo Memory）

定义：

- 针对某个项目/代码仓库/场景的持续性记忆：
  - 项目约定（coding style、目录结构、模块职责）。  
  - 已完成任务的要点和决策摘要。  
  - 「坑点」和 known issues（例如：某组件易出 bug 的地方）。

工程要点：

- 存储形态：
  - 结构化文件（如 `memdir` 风格的 Markdown/JSON）。  
  - 结合向量检索的「项目记忆库」。
- 更新策略：
  - 每次任务结束时，从 session 中提炼一小段 summary 写入项目记忆。  
  - 定期整理和去重，避免无限膨胀。

## 3.3 长期用户 / 团队记忆

定义：

- 跨项目、跨场景的长期偏好与习惯：
  - 用户/团队的编码风格偏好。  
  - 文档书写习惯、沟通语气。  
  - 经常性的「黄金原则」和「禁忌模式」。

工程要点：

- 严格的隐私与授权：
  - 明确哪些信息可以长期保存，哪些必须只存在于当前会话。  
  - 支持用户查看与清理自己的长期记忆（类似「隐私中心」）。
- 对记忆内容进行抽象与压缩：
  - 避免直接保存完整对话，而是提炼成规则或偏好描述。  
  - 便于在不同场景中复用。

# 4. 记忆写入：从「日志」到「升级」

记忆不等于简单记录所有对话或操作日志，需要有明确的「升级」路径：

1. 观察层：
   - 收集对话历史、工具调用、任务结果、用户反馈。  
2. 筛选层：
   - 使用规则/Agent 从中挑选「值得记住的内容」。  
   - 例如：
     - 用户明确声明的偏好变更（"以后都用中文回答"）。  
     - 重要决策（「这个项目约定都使用 hexagonal architecture」）。  
     - 反复出现的问题及其解决方案。
3. 提炼层：
   - 将原始内容提炼为结构化条目：
     - `type: preference/decision/issue`  
     - `scope: session/project/user/team`  
     - `text: ...`  
4. 落盘层：
   - 写入对应层级的记忆存储（session store、项目 memdir、用户 profile 等）。

在 Claude Code 中，这条路径部分体现为：

- 使用 hooks / tasks 在任务结束时更新 `memdir`。  
- 通过 session guidance 和 memory prompt 在新任务启动时注入相关记忆。

# 5. 记忆读取与注入策略

## 5.1 注入位置与优先级

- 将结构化记忆注入到 system prompt 的特定 section，而不是混杂在普通对话中：
  - 例如：
    - 「用户/团队偏好」section。  
    - 「项目约定与背景」section。  
    - 「当前任务的历史结论」section。

- 配合 prompt cache boundary：
  - 将相对稳定的记忆放在可缓存区域，减少重复 token 成本。  
  - 将高度上下文相关的记忆（如当前任务 summary）放在动态区域。

## 5.2 检索与过滤

- 对项目/长期记忆使用检索机制：
  - 向量检索 + 关键词过滤，从记忆库中选出与当前任务最相关的条目。  
- 对不同任务类型使用不同的检索策略：
  - 代码重构 vs 文档撰写 vs 需求分析，会需要不同的记忆片段。

## 5.3 冲突与过期

- 冲突处理：
  - 若记忆中存在冲突（旧规则 vs 新规则），需要明确优先级：
    - 例如：最新记忆覆盖旧记忆。  
    - 或要求人工/Agent 进行记忆清理。

- 过期机制：
  - 为部分记忆设置 TTL 或「软过期标记」，避免长期保留已经无效的信息。  
  - 定期运行「记忆巡检 Agent」，清理无用或过期条目。

# 6. 安全与隐私约束

记忆工程天然涉及隐私和合规风险，需要明确的边界：

- 数据分级：
  - 将记忆内容按敏感度分级（公开/内部/敏感/机密）。  
  - 对不同级别采用不同的存储与访问策略（如加密、访问审计）。

- 用户控制权：
  - 提供接口让用户查看、纠正和删除自己的长期记忆。  
  - 对于团队记忆，需有明确的管理员角色负责治理。

- 最小持久化：
  - 仅存储对长期行为真正有价值的信息。  
  - 避免存储完整原始对话或敏感 payload。

# 7. 与 Prompt / Context / Harness 工程的关系

- 与 Prompt Engineering：
  - 记忆决定「有哪些稳定信息要写进 Prompt」，Prompt 决定「如何表达这些记忆」。  
  - 例如：将团队约定转化为 system prompt 中的明确指令。

- 与 Context Engineering：
  - 上下文工程决定「在这一轮调用中需要哪些事实」。  
  - 记忆工程提供候选事实来源，并确保这些事实可被检索。  

- 与 Harness Engineering：
  - 记忆写入/读取本身应受到驭控约束：
    - 哪些操作允许修改记忆。  
    - 在什么条件下可以自动更新项目/用户记忆。  
  - 对记忆相关的演化行为（例如自动调整偏好）需要 Eval 与审计。

# 8. 在 AIAgent 中的使用建议

1. 在架构层面显式设计记忆组件：
   - 不要将记忆逻辑散落在业务代码或临时变量里。  
   - 将记忆视为一等公民组件，具有清晰的 API 与数据模型。

2. 为每个场景定义记忆策略：
   - 对每个 `scene-*/`：
     - 明确该场景需要哪些层级的记忆（仅 session / project / user）。  
     - 明确记忆的写入/读取触发点（任务结束、用户修改偏好等）。

3. 在实现上优先从「精简记忆」做起：
   - 一开始只记录少量关键偏好与决策。  
   - 随着项目演进，通过真实反馈逐步扩展记忆内容。

4. 与评估体系结合：
   - 对「有记忆」与「无记忆」的模型行为进行对比评估，确保记忆带来正收益。  
   - 对记忆引入的潜在负面影响（强化错误偏好、泄露旧信息）进行专门测试。

# 1. 客户业务目标与技术约束

- 客户业务背景：
  - 多租户 B2B SaaS 平台，面向中大型企业客户提供在线 CRM/项目管理/协同办公等服务；
  - 平台功能复杂、配置灵活（对象模型、字段、流程、权限高度自定义），企业管理员和普通用户的学习成本高。
- 业务目标（示例）：
  - 将新租户从注册到可用（Time-to-Value）时间缩短 30%；
  - 将平台内「配置错误导致的故障/工单」量降低 20%；
  - 管理员/开发者与平台支持团队之间的咨询量下降 30%；
  - 提升核心模块的使用率和留存率（按月活租户数、关键功能使用次数衡量）。
- 业务边界与不做的事：
  - 不直接对租户生产数据执行高危变更（删除/批量更新），只生成配置变更草案和脚本，由管理员确认；
  - 不在早期版本中引入「自动化计费变更」类能力，所有计费相关操作保留人工审批；
  - 不尝试替换现有客服/工单体系，而是与之集成。
- 技术与组织约束：
  - SaaS 平台已经在云上运行，有成熟的多租户数据库、配置中心、监控和审计；
  - 技术栈：后端以 TypeScript/Go 为主，前端 Web + 移动，已有 Kubernetes 集群；
  - 有基础 MLOps/LLMOps 能力，但不希望自建大规模推理集群，优先使用云模型 + 轻量网关；
  - 合规：需要严格的租户隔离、操作审计与数据脱敏，不允许将敏感字段以明文形式暴露给 LLM。

# 2. 场景与用例拆解

- 场景定义：
  1. 租户管理员配置助手（Admin Config Assistant）；
  2. 开发者 API 集成与调试助手（Dev Integration Assistant）；
  3. 运维/运营监控与告警分析助手（Ops/FinOps Assistant）。

- 典型用例清单（示例）：
  - 租户管理员：
    - 「我想管理销售线索和客户」→ 自动生成对象/字段/流程配置草案；
    - 「帮我检查当前权限配置是否合理」→ 输出风险点和建议（仅建议）；
    - 「为新事业部复制一套配置」→ 输出配置差异与迁移脚本草案。
  - 开发者：
    - 粘贴错误日志/代码片段 → Agent 定位签名/权限/参数等问题，并给出修复建议和调用示例；
    - 「如何在 Node/Go/Java 里调用这个 API」→ 生成示例代码和最佳实践；
  - 运维/FinOps：
    - 「最近三个月这个租户的用量异常吗？」→ 生成用量分析和成本建议；
    - 「为什么这个租户本月账单比上月高很多？」→ 分析变动原因。

- 优先级与波次划分：
  - v1（MVP）：
    - 文档/配置说明 RAG 问答；
    - API 调用示例与错误诊断建议；
    - 只读监控与账单分析（不执行写操作）。
  - v2：
    - 管理员配置检查与配置草案生成（需人工确认）；
    - 简单自动化脚本草案（仅测试环境自动执行）。
  - v3：
    - 在严格审批机制下，对于低风险操作支持有限自动执行（例如创建只读报表/视图）。

# 3. 模型与 MaaS 能力选型

- 模型访问方式（MaaS 方案）：
  - 通过自建 OpenAI 兼容模型网关，将 SaaS 平台与多家云模型厂商解耦：
    - 初期以 OpenAI + Qwen 为主；
    - 后续可引入 Anthropic / Gemini / Llama 等模型。
  - 理由：
    - 统一鉴权、限流、日志与审计；
    - 支持根据场景动态切换模型，避免硬绑定某一厂商；
    - 便于在同一接口下混用云模型与（未来）自托管模型。

- 模型家族与规格选择（初稿）：
  - 对话/推理主模型：
    - 高价值链路（复杂配置建议、运维策略建议）：GPT-5.x / Claude Sonnet；
  - 低成本模型：
    - FAQ/文档问答、简单错误解释：GPT-5.3 Instant / Qwen3.5-Plus；
  - Embedding 模型：
    - 一致使用 OpenAI / Qwen embedding（视费用与中文表现而定）。

- 多模型路由策略：
  - 根据「用例价值+复杂度」选择不同模型，并在网关层实现：
    - FAQ/文档问答 → 小模型；
    - 涉及配置/权限/计费分析 → 大模型；
  - 保留 A/B 测试能力：
    - 在部分租户或流量上对比不同模型的效果和成本，逐步固化策略。

# 4. 工程链路设计：推理 / 微调 / RAG / Agent

## 4.1 推理链路（Inference Pipeline）

- 请求路径：
  - SaaS 控制台前端 / 内嵌 Chat → 平台 API 网关 → AI Gateway → MaaS（多厂商模型）→ 返回流式结果；
  - 内部系统（监控/计费）通过内部 API 调用 AI Gateway 提供的分析接口。

- Prompt & Context 架构：
  - System Prompt 模块：
    - 身份与角色（平台智能顾问/配置助手/Dev 助手）；
    - 做事规则（不自动执行高危操作、不捏造配置、不越权访问租户数据）；
    - 工具使用规则（优先使用平台封装的 API 工具，不直接构造 SQL 等）。
  - 上下文：
    - 租户级配置快照（对象/字段/流程/权限）；
    - 当前用户角色与环境（prod/stage）；
    - 相关文档和历史对话摘要。

## 4.2 微调 / 指令微调

- 初期：不做自有微调，优先利用高质量 Prompt + RAG；
- 中期：视数据和需求考虑指令微调：
  - 目标：提升配置建议、错误诊断和账单分析的稳定性与专业度；
  - 数据来源：
    - 历史对话 + 人工标注的「好答案」；
    - 平台文档 + 内部 Runbook；
  - 使用云厂商提供的指令微调服务，版本管理在模型网关侧记录。

## 4.3 RAG 与知识/数据接入

- 数据源：
  - SaaS 平台文档（产品说明、配置指南、API 文档、Runbook）；
  - 配置元数据（对象模型、字段定义、权限策略）；
  - 监控与计费报表的结构化摘要。

- 向量库与索引：
  - 早期：Postgres + pgvector（随平台数据库部署）；
  - 规模上升后：迁移到 Qdrant / Milvus，按租户/模块分 Collection；
  - 索引结构：
    - 按文档类型/模块/版本分区；
    - 保留引用信息（文档 ID，版本号）。

- 检索策略：
  - 按租户/环境过滤；
  - FAQ/文档问答：直接取若干 top-k 片段组成回答上下文；
  - 配置建议：组合租户配置快照 + 相关文档片段。

## 4.4 Agent 编排与工具层

- Agent 角色划分：
  - `AdminAgent`：围绕租户配置、权限、策略提供建议与检查；
  - `DevAgent`：围绕 API 集成、SDK 使用、错误排查提供示例和建议；
  - `OpsAgent`：围绕监控、告警、扩容、计费分析提供洞察；
  - `RouterAgent`：根据意图路由给不同子 Agent。

- Orchestration 框架：
  - 顶层：自研 orchestrator（基于现有任务系统/事件总线）处理多租户、限流、权限；
  - 子流程：在复杂交互（如多步排查、配置修改建议）内部使用 LangGraph 建模 Agent 流程。

- 工具层与驭控：
  - 工具清单：
    - 平台配置查询工具、权限查询工具、API 调试工具、监控查询工具、计费查询工具等；
  - 高危工具（写配置/执行脚本）只在测试环境开放自动执行，生产环境仅输出脚本草案 + 操作说明；
  - 工具执行链：输入校验 → PreHook（权限/合规检查）→ 权限决策（allow/ask/deny）→ 执行 → PostHook（记录与审计）。

# 5. 效果评估体系与测试策略

## 5.1 指标体系

- 业务指标：
  - 新租户启用时间（从注册到首个业务流程上线）；
  - 与配置/集成相关的支持工单量与解决时长；
  - 租户留存率、关键功能使用率。
- AI/模型指标：
  - 建议采纳率、正确率（人工审查样本）；
  - 一轮解决率（无需再次追问）；
  - 工具调用合规率（无越权/违规调用）。
- 系统指标：
  - 调用延迟、错误率；
  - 模型调用成本（按租户/模块维度）。

## 5.2 测试与评估策略

- 离线评估：
  - 构建覆盖主要场景的 Eval 集（配置建议、API 诊断、计费分析）；
  - 使用规则 + Judge LLM + 人审结合评估回答质量与安全性。
- 回归与变更管理：
  - Prompt/模型/工具/编排变更前后，在 Eval 集上运行回归，对比行为差异；
  - 在 CI/CD 中增加关键用例自动评估步骤。
- 在线监控：
  - 记录每次建议的结果与人工采纳情况；
  - 对高风险建议和异常模式进行抽样人工复审。

# 6. 上线路径与部署策略

- 环境与灰度：
  - 先在内部租户和部分试点客户中开启（白名单）；
  - 初期只读模式 + 建议模式，不执行写操作；
  - 根据表现逐步拓展租户范围和自动化程度。

- 安全与合规：
  - 所有模型调用参数中敏感数据脱敏；
  - 工具访问严格按租户隔离；
  - 对所有高风险建议和脚本草案保留审计日志与回放能力。

- 运维与运营：
  - 为平台运营/CS 团队提供「AI 建议效果」监控面板；
  - 定期复盘各租户的使用情况和价值实现程度，指导下阶段迭代。

# 7. 风险分析与演进路线

- 已知风险：
  - 配置建议错误可能带来安全/合规/成本风险（虽然不自动执行，但易被误采纳）；
  - 对特定行业/租户的语义理解不充分，导致建议不匹配；
  - 模型供应商价格/策略变化带来的成本波动。

- 缓解策略：
  - 对高风险建议强制二次确认与审查；
  - 引入租户/行业上下文的更多结构化信息，提高理解能力；
  - 模型网关支持多模型、动态路由与熔断预案。

- 演进路线：
  - MVP：FAQ + 文档问答 + 简单 API 诊断（只读）；
  - Beta：引入配置检查与脚本草案生成（测试环境可自动执行）；
  - GA：部分低风险操作自动执行 + 完整评估与审计体系。
